First, thirty seconds of background for anyone new to specification-driven development.
[pause 0.35]
Instead of prompting an AI one request at a time, you write a specification defining software behavior. Agents build software from the specifications.
[pause 0.35]
There are many approaches to creating specifications, including simply asking your LLM to create one.
[pause 0.35]
The promise is that your software can get better as you update the specifications, but there are two gaps.
[pause 0.35]
The first gap comes from specification size. Output quality is related to prompt size and complexity.
[pause 0.35]
With a small spec: great results. But with a big spec, software quality falls off a cliff.
[pause 0.35]
Better results occur when you divide the software into small units of work.
[pause 0.35]
Gap two is reproducibility.
[pause 0.35]
The LLM is not deterministic. Large specifications drift. That means you can't reproduce.
[pause 0.35]
The solution, surprisingly, is using software development best practices.
[pause 0.35]
For 25 years, software engineers have been decomposing large projects using Agile. Agile is a communication process whereby the team agrees on small steps or stories that will deliver the product owner's vision.
[pause 0.35]
Test-driven development focuses on build quality. Testable assertions define exact software behavior and edge cases.  TDD is a huge quality gate for reproducible builds.
[pause 0.35]
These two well-established processes enable builds to be reproducible and therefore allow them to serve as the source of truth for your projects.
