# Drydock Acceptance Criteria

| Field       | Value |
|-------------|-------|
| Version     | 20260610 V1 |
| Description | High-level acceptance criteria for declaring Drydock delivered. Product-owner memory aid; not a build artifact. |

Positive, testable outcomes. Drydock is delivered when every box is checked.

## The Three That Convert

- [ ] **The ten-minute demo.** A recorded end-to-end run on a real project: blueprint → plan →
  build → QuarterDeck decision → incremental rebuild. The loop running is worth more than the
  entire written specification.
- [ ] **The Ship's Log → blog pipeline.** A public design-decision trail generated by Drydock
  itself — a self-updating portfolio in the product owner's voice.
- [ ] **The Spec Kit import.** `drydock import --format speckit` consumes a real public Spec Kit
  project and produces a reviewable Blueprint. The demonstration that makes Drydock legible to
  people who already know the category.

## Command Surface

- [ ] No deferred stubs remain; every Blueprint command is implemented and tested.
- [ ] `drydock plan create <Blueprint> <Target>` generates and maintains the target
  `MANIFEST.md`, including cost-reducing work grouping and the `PROMPT_WARN_TOKENS` oversize warning.
- [ ] `drydock build`, `build status`, and `build score` execute the frontier, report state, and
  produce `SCORECARD.md`.
- [ ] The QuarterDeck serves evidence, questionnaires, and the sprint board; approve, revise,
  reject, and reopen-failed all write back through the single decision writer.
- [ ] `drydock refit` updates Blueprint and target together and appends rationale to the
  Ship's Log.
- [ ] `drydock analyze` reports drift, open questions, and specification changes not covered by a
  Ship's Log entry.
- [ ] `drydock rigging compact | update | verify` propagate and check the governance contract.
- [ ] `drydock document generate | assemble` produce versioned `docs/index.html`.
- [ ] `drydock import --format auto | source | speckit` produce a proposed Blueprint with a
  conversion report.

## Distribution

- [ ] Package published on PyPI under the reserved name; `pip install drydock` and
  `uv tool install drydock` work from a clean machine.
- [ ] Installed-wheel behavior verified: Rigging resources resolve from the packaged copy, not the
  source tree.
- [ ] CI green on GitHub Actions: full test suite, lint, format, and the coverage threshold.
- [ ] Tagged releases with `CHANGELOG.md` entries; version reported by `drydock --version`.

## Proof of Methodology

- [ ] One real project delivered end-to-end from Blueprint to accepted software entirely through
  the Drydock workflows and the QuarterDeck.
- [ ] Incremental rebuild proven: edit one specification file; only stale blocks rebuild,
  accepted work is untouched.
- [ ] A failed story recovered entirely through the QuarterDeck reopen path — no hand-edited
  plan files.
- [ ] Both subscription providers (`claude`, `codex`) complete the same build.
- [ ] Prototyper workflows fully ported; Prototyper archived as read-only V1 reference.
- [ ] Typed Specification contract changes propagated to `Rigging/SPECIFICATION_CONTRACT.md`,
  templates, and validation.

## Communication

- [ ] Drydock whitepaper published on the Web Cloud Studio site.
- [ ] Drydock's own documentation generated by `drydock document` and published.
- [ ] README quickstart verified start-to-finish on a machine that has never seen the repository.
- [ ] First Ship's Log–derived design post published.
