Specification Driven Design Methodology

Specification Driven Design

A methodology for building software from structured specifications using AI agents. One specification set produces one working prototype — no ambiguity, no drift.
Ed BarlowWeb Cloud Studio2026

The Problem

AI code generation works well for small, self-contained tasks. But building a complete application — with a database, multiple screens, shared UI patterns, and operational concerns — requires more than a single prompt. Without structure, AI builds diverge from intent, accumulate inconsistencies, and become harder to iterate on with each cycle.

The Approach

Specification Driven Design separates what to build from how to build it. The developer writes concise, structured specification files. An opinionated technology stack and a set of business rules provide the how. A build script assembles everything into a single prompt that an AI agent executes in one pass.

Specification Files

Each project is defined by a small set of markdown files, each with a specific role:

  • METADATA.md — identity, stack, status (the service catalog entry)
  • ARCHITECTURE.md — modules, routes, directory layout
  • DATABASE.md — tables, columns, types (schema only)
  • FUNCTIONALITY.md — what the application does at a high level
  • SCREEN-*.md — per-screen: route, layout, interactions
  • FEATURE-*.md — per-feature: trigger, sequence, reads, writes
  • UI-GENERAL.md — shared UI patterns across screens

Technology Rules

A shared rules engine (CLAUDE_RULES.md) defines agent behavior: file structure, naming conventions, error handling, security, commit standards. Stack files (flask.md, sqlite.md, etc.) provide copy-paste implementation patterns. The AI agent follows these rules exactly — no creative interpretation.

One-Shot Build

The oneshot.sh script validates the specification, concatenates all inputs (specification files + rules + stack patterns), and produces a single prompt. The AI agent reads this prompt and builds the entire application in one pass. The result is tagged in git so every build is traceable to the exact specification state that produced it.

Iteration

After the initial build, changes flow through the specification — not through ad-hoc code edits. Numbered tickets (SCREEN-NNN-*.md, FEATURE-NNN-*.md, PATCH-NNN-*.md) describe focused changes. The iterate.sh script assembles these into an iteration prompt that the AI agent applies to the existing codebase.

Quality tools keep things on track: scorecard.sh measures specification-to-code alignment, spec_iterate.sh identifies specification gaps, and tran_logger.sh extracts bugs and ideas from AI session logs.

Promotion

When a prototype is ready, merge.sh squash-merges the feature branch into the base branch. ProjectValidate.sh checks compliance against maturity-level rules (IDEA → PROTOTYPE → ACTIVE → PRODUCTION). ProjectUpdate.sh propagates the latest rules and templates to promoted projects.
