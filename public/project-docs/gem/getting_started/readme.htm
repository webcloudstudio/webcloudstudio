Getting Started Guide

for the GEM

This guide will give information on how to get started with the Generic Enterprise Manger (GEM). This document describes the structure of the code line and introduces you to the basic concepts and describes how the application works.

GEM provides a framework to manage multiple database systems. The system is built upon a set of common perl libraries. The concept is that the various monitoring and administration programs provided with gem be simple, clear, documented, and small. To implement this concept, the majority of the work is relegated to the library modules. One side benefit of this architecture, is that you can copy the monitoring code line and change it, customizing it for your environment. These programs, by and large, are small and this kind of customization is fairly easy if you understand perl.

GEM is designed for simplicity. The goal is to put code, data, and libraries in locations where a reasonable administrator would expect them to be found. After you extract GEM, please poke around and familiarize yourself with the product. Look for utilities in the ADMIN_SCRIPTS subdirectory and configuration files in the conf subdirectory. Knowledge of the layout is essential to maintaining and operating GEM as your production server managment interface.

This document provides an overview of GEM layout.

Directory Structure

The first thing to understand is the basic directory structure of the installation. The command line programs that implement most of the functionality exist in the ADMIN_SCRIPTS subdirectory, which is further subdividied by package. Configuration files are stored in the conf subdirectory. Data and formatted html reports is stored in the data subdirectory. Three examples of data stored in the data subdirectory are the GEM File Repository (where we keep collected files and system log files), the GEM Console (in CONSOLE_REPORTS) where we keep static html report output, and the historical information about your servers. The main GEM programs (gem.pl, troubleshoot.pl, and configure.pl) are in the the root directory. The rest of the heirarchy is for support files.

GEM Scheduled Tasks

One of the core features of GEM is its monitoring and maintenance scripts. These scripts perform the systems monitoring and alarming and provide useful routine maintenance activities. The scripts are customized to your environment by the GEM configuration manager. Some of these scripts user your configuration file, for example several batch jobs are created on a server by server basis.

The gem.pl program will create scheduled tasks for these jobs using your Windows scheduler, and will create a Unix crontab file for jobs you need to run on Unix. Each batch job is basically a call to a command line scripts located in the ADMIN_SCRIPTS directory. Output is redirected to error files in a standard location. GEM creates a set of .bat and .ksh driver scripts that have obvious names to be the scripts that you actually run or schedule.

Batch Job scripts are stored in the win32_batch_scripts and Unix_batch_scritpts directories. Within those directories are 4 subdirectories as follows:

GEM Base
unix_batch_scripts This is where .ksh scripts that run under Unix exist. Output redirected.
interactive Output not redirected
batch Scheduled scripts. Output redirected.
plan_interactive Scripts for backups. Output not redirected.
plan_batch Scripts for backups. Output redirected.

The above tree is duplicated in the win32_batch_scripts subdirectory.

Each script is created as a .ksh on Unix and as both a .bat and .ksh script on Windows. Copies of the script exist in an interactive and batch directory. The batch scripts redirect output to locations in the data tree (so you cant see output when run interactively) but are identical to the interactive scripts. When you run a scheduled task, the version in the batch directory is what is scheduled. When you wish to run a program interactively, you should use the interactive version - because you want to see output. The plan_* directories store pregenerated maintenance plan scritps. The plan directories were separated from the regular directories because the tree becomes hard to navigate when you get large numbers of servers (and a large number of maintenance plans).

Backup and Maintenance Plans

GEM offers a maintenance plan based system maintenace facility.  Plans are set up in the file configure.cfg.  All normal features related to systems maintenance are covered including Backups, DBCC, Update Statistics, and Log Shipping.  Additionally, backup compression for SQL Server, notification, and extensive reporting are provided.  The command line interface provides a level of flexibility not available with native SQL Server facilities. These scripts can be run on all versions of sybase & sql server, but for routine backups, they are best suited for sybase and sql server 2000 only, as the native log shipping and mirroring in sql server 2005 and 2008 work well. If you want to reorg a database or run update statistics, these scripts provide value for all versions.

GEM Post Install Tasks Manager

The GEM Enterprise Manager (gem.pl) is the simple user interface for GEM post installation tasks. Basically, it calls other perl programs with the appropriate command line paths. These tasks might be to rebuild your batch jobs or to reschedule them.

Documentation and Diagnostics

All system documentation uses the standard perldoc format. Perldoc is a utility provided with perl to extract documentation from programs. To get full documentation on a program, simply run perldoc program. The majority of the GEM documentation is composed from the embedded perldoc. You can see a short version of the help by simply running perl program.pl -?, which will provide useful syntax help. 

Older database programs use the standard syntax of -Uuser -Ppassword -Sserver -Ddatabase for arguments. Newer programs use --USER=username --PASSWORD=password --SERVER=servername --DATABASE=database convention.

If you have problems with a programs, try running with -d or --DEBUG flags set. These will run the program in verbose/diagnostic mode and the program will print useful diagnostic information. This is the standard way of debugging problems.
