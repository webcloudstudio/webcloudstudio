A phone app sends a voice note at 2 AM. A cron job on another machine finishes and wants to notify the platform. A script fails and wants to enqueue a retry. These producers need a queue that accepts work regardless of whether the consumer is alive.\n\n**File-based, dead simple:** The queue is a JSONL file. Producers append a line. That is the entire write path \u2014 no server, no socket, no protocol. Any language that can append to a file can enqueue work. When GAME starts (or on demand), it drains the queue: reads each line, dispatches to the appropriate handler, and marks it processed.\n\n**Why not a real message broker:** This platform is for a single developer or a small team running local services. RabbitMQ or Redis Streams are correct at scale but wrong here \u2014 they add infrastructure, configuration, and failure modes that a JSONL file does not have. If the developer outgrows file queues, the service interface stays the same and the transport changes to a broker behind the config.\n\n**The VoiceForward example:** Today VoiceForward (FEATURE-VOICEFORWARD.md) posts directly to the GAME server via `POST /api/voice/upload`. If the server is down, the recording is lost. With AsyncQueue, the phone app (or a lightweight relay) writes a queue message. When GAME starts, it drains the queue, transcribes the audio, and appends the text \u2014 no recording lost, no timing dependency.\n\n**The general pattern:** Any service that benefits from \"submit now, process later\" can use AsyncQueue as a transport. The service manifest (FEATURE-ServiceInterfaces.md) declares `async: true`, and the platform generates the queue file and drain logic automatically.\n\n---\n\n## Queue File Format\n\nEach queue is a JSONL file at `data/queues/{queue-name}.queue.jsonl`. One line per message:\n\n```jsonl\n{\"id\":\"a1b2c3\",\"queue\":\"voice\",\"service\":\"voiceforward\",\"tool\":\"transcribe\",\"payload\":{\"audio_path\":\"/tmp/rec_001.webm\",\"label\":\"BOOK IDEAS\"},\"submitted_at\":\"2026-04-06T02:15:00Z\",\"status\":\"pending\",\"priority\":\"normal\"}\n{\"id\":\"d4e5f6\",\"queue\":\"voice\",\"service\":\"voiceforward\",\"tool\":\"transcribe\",\"payload\":{\"audio_path\":\"/tmp/rec_002.webm\",\"label\":\"GAME SPEC\"},\"submitted_at\":\"2026-04-06T02:17:00Z\",\"status\":\"pending\",\"priority\":\"normal\"}\n```\n\n### Message Fields\n\n| Field | Type | Required | Description |\n|-------|------|----------|-------------|\n| `id` | string | Yes | Unique message ID (UUID or short hash) |\n| `queue` | string | Yes | Queue name (determines which file and which drain handler) |\n| `service` | string | Yes | Target service name from the service registry |\n| `tool` | string | Yes | Tool name within the service to invoke |\n| `payload` | object | Yes | Tool inputs (must match the tool's input schema) |\n| `submitted_at` | string | Yes | ISO 8601 timestamp |\n| `status` | string | Yes | `pending` / `processing` / `done` / `error` / `expired` |\n| `priority` | string | No | `low` / `normal` / `high` / `critical` (default: `normal`) |\n| `ttl_seconds` | integer | No | Message expires if not processed within this many seconds after submission |\n| `result` | object | No | Set by drain handler on completion |\n| `error` | string | No | Set by drain handler on failure |\n| `processed_at` | string | No | Timestamp when processing completed |\n\n---\n\n## Queue Operations\n\n### Submit (Producer Side)\n\nProducers write to the queue by appending a JSONL line. Three ways to submit:\n\n**1. Direct file append (works without GAME running):**\n\n```bash\necho '{\"id\":\"'$(uuidgen)'\",\"queue\":\"voice\",\"service\":\"voiceforward\",\"tool\":\"transcribe\",\"payload\":{\"audio_path\":\"/tmp/rec.webm\",\"label\":\"BOOK IDEAS\"},\"submitted_at\":\"'$(date -Iseconds)'\",\"status\":\"pending\",\"priority\":\"normal\"}' >> data/queues/voice.queue.jsonl\n```\n\n**2. Python helper in common.py:**\n\n```python\nfrom common import services\nservices.async_queue.submit(\n    queue=\"voice\",\n    service=\"voiceforward\",\n    tool=\"transcribe\",\n    payload={\"audio_path\": \"/tmp/rec.webm\", \"label\": \"BOOK IDEAS\"}\n)\n```\n\nThe helper generates the ID, timestamp, and appends to the file. Works without GAME running.\n\n**3. REST API (requires GAME running):**\n\n```\nPOST /api/services/async-queue/submit\n{\n  \"queue\": \"voice\",\n  \"service\": \"voiceforward\",\n  \"tool\": \"transcribe\",\n  \"payload\": {\"audio_path\": \"/tmp/rec.webm\", \"label\": \"BOOK IDEAS\"}\n}\n```\n\n### Drain (Consumer Side)\n\nDraining processes pending messages in priority then submission order:\n\n1. Read the queue file\n2. Filter for `status = pending` (skip done/error/expired)\n3. Check TTL \u2014 mark expired messages as `expired`\n4. For each pending message (highest priority first, then oldest first):\n   a. Set status to `processing`\n   b. Resolve the target service and tool from the service registry\n   c. Call the tool (via internal Python dispatch, not REST \u2014 no network dependency)\n   d. On success: set status to `done`, record `result` and `processed_at`\n   e. On failure: set status to `error`, record `error` message\n5. Rewrite the queue file with updated statuses\n\n**Drain triggers:**\n- **Startup:** GAME drains all queues on startup (after scan completes)\n- **On demand:** `POST /api/services/async-queue/drain` or `game-cli async-queue drain`\n- **Periodic:** Optional scheduler job (e.g., drain every 5 minutes)\n- **File watch (future):** inotify/fswatch on the queues directory for immediate processing\n\n### Queue Rotation\n\nProcessed messages (done/error/expired) accumulate in the queue file. Rotation moves them to an archive:\n\n- After drain, if the queue file exceeds 1000 lines or 1 MB, rotate:\n  - Move completed messages to `data/queues/archive/{queue-name}.{date}.jsonl`\n  - Rewrite the active file with only pending/processing messages\n- Archive files are kept for 30 days (configurable), then deleted\n\n---\n\n## Service Manifest\n\n```yaml\nname: async-queue\ndescription: File-based store-and-forward message queue \u2014 submit work when the server is down, drain on startup\nversion: 1.0.0\n\ntransports:\n  rest: true\n  cli: true\n  mcp: false\n  async: false    # The queue itself is not async-backed (it IS the async mechanism)\n\ntools:\n  - name: submit\n    description: >\n      Submit a message to a named queue. Today GAME exposes capabilities through REST endpoints, bash scripts (CLI), and the web UI. Adding MCP and async queues creates five transports for the same underlying capabilities. Without a unifying descriptor, each transport is a silo.\n\n**The core idea:** every service is described using an MCP-shaped manifest \u2014 name, description, and typed tool definitions \u2014 but the service does NOT speak the MCP protocol unless it is an actual MCP server. The manifest is a **metadata format**, not a transport. GAME reads these manifests to populate the Service Catalog, generate CLI wrappers, expose REST endpoints, and optionally start real MCP servers.\n\n**Why MCP-shaped:** MCP already solved the \"how do you describe a tool so an AI agent can call it\" problem. By adopting the same shape (name, description, input schema, output schema) for ALL services \u2014 not just MCP ones \u2014 we get a uniform interface that works for humans (Service Catalog UI), scripts (CLI gateway), agents (MCP or REST), and async consumers (queue). Developers learn one descriptor format. Claude reads one format. The platform renders one format.\n\n**What this enables:**\n- A developer writes a service manifest once. GAME exposes it via whichever transports are enabled.\n- Projects declare service dependencies in `services.yaml`. `common.py` reads this config and dispatches calls to the correct transport without the application knowing how the service is hosted.\n- Swapping implementations (e.g., platform logging vs. external logging) is a config change, not a code change.\n- The same descriptor that tells Claude what tools are available also tells the application code at runtime how to call them. One source of truth for both build-time (agent) and run-time (application).\n\n**What this is NOT:** This is not a mandate that all services speak MCP protocol. stdio MCP is one transport option. That board uses a fixed set of states (idea \u2192 proposed \u2192 ready \u2192 in_development \u2192 testing \u2192 done) tied to `spec_tickets`. The Workflow Service generalizes this into a platform service that any project can use for any kind of state machine \u2014 not just specification tickets.\n\n**One interface, many workflows:** A deployment pipeline, a content review process, a bug triage queue, and the existing specification ticket lifecycle are all state machines. They differ in their states, transitions, and payloads \u2014 not in their fundamental operations (create, transition, query, list). The Workflow Service provides those operations as a generic interface. The Kanban screen reads from the same data through the workflow service API. No migration needed \u2014 the existing states map directly to a workflow template.\n\n**Integration with other services:** The workflow service is a state machine, not a task executor. When a transition fires (e.g., specification ticket moves to \"in_development\"), the workflow service emits an event. The workflow service does not run scripts \u2014 it manages state.\n\n---\n\n## Service Manifest\n\n```yaml\nname: workflow\ndescription: Generic state machine for project task and process lifecycle management\nversion: 1.0.0\n\ntransports:\n  rest: true\n  cli: true\n  mcp: true\n  async: false\n\ntools:\n  - name: create\n    description: >\n      Create a new workflow instance from a template.
