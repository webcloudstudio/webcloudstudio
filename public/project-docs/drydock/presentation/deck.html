The gap

Spec-driven design forgot
25 years of software best practices.

  • It gave us specifications — and threw away the process that ships them.
  • You can refine a spec forever. Without process you still can't reproduce the build.
  • The result: drift, lost context, and a thousand lines no one can govern.

As any software engineer knows — a result you can't reproduce isn't a result.

The insight

Reproducible builds need a process.
That process already exists.

Process, not just specs

You can refine a specification. You can't reproduce a build without control and a repeatable process.

That process is Agile

25 years of best practices, codified in the Agile Manifesto — how developers ship at major companies.

Drydock is Agile applied to AI SDD delivery. The specification stays the single source of truth.

Introducing Drydock

SAIL - Governed, Blueprint-driven delivery system.

S · SET UP A · ANALYZE I · IMPLEMENT L · LOOP

SAIL is an agile process for software delivery

  • **Setup** your system and import your notes or LLM-created specifications.
  • **Agile/Analyze** decomposes into features/stories/spikes and defines a dependency graph.
  • **Implement** builds working software using a context optimized process.
  • **Loop** and Maintain using change tickets or edits to your specifications.



SAIL also creates documentation and measures build quality and drift.

Drydock Rigging lets builds use common stack files and branding.

SA · ANALYZEIL

Charting the course

Sources→ import→ analyze→ ANALYSIS.md
  • Import markdown, source code, Spec Kit, or loose notes.
  • Decompose into stories, acceptance milestones, blockers, questions — using agile.
  • If the LLM finds blockers → it stops and asks. You answer in the QuarterDeck and rerun.
  • The output is a tentative set of Features, Stories, and Spikes for you to review in the QuarterDeck.
The core differentiator

The QuarterDeck is your Agile Web interface to the process.

An optimized communication path between the Commander and the LLM Process.

  • Analyze creates the story hierarchy, identifies blockers, and surfaces questionnaires for review.
  • Answer and redirect or proceed. Your feedback is persisted.
  • The Commander controls — The GOAL is Working Software you can Iterate.
  • drydock rigging compact produces minimal context using Builder/User roles for each file
The engine

The Manifest is a dependency graph of your build.

  • drydock plan turns the analysis into typed Blueprints + the Manifest.
  • The LLM estimates Story Points (token cost) for every story using the dependency graph.
  • Drydock stacks the minimal context of the correct files into build prompt.

Context is engineered, not hoped for. That's how builds become reproducible.

SAI · IMPLEMENTL

IMPLEMENT - Sailing the frontier

  • drydock build walks the graph and builds the runnable frontier. The build is done by one context optimized step at a time.
  • Every step can be verified against acceptance criteria.
  • drydock build score measures delivery health across seven dimensions — spec completeness, test coverage, drift, and more.

SAIL · LOOP

The refit

Software is never done. drydock refit changes the application while keeping the Blueprint and the code aligned.

  • Option 1: Edit your specification files. Drydock tracks git commits for every built file and can do a minimal rebuild of only what changed.
  • Option 2: Change tickets tied to existing specification files, dynamically built using minimized context.
  • Summary: The specification is the source of truth. Minimal drift.
